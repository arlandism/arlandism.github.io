A "spike" is when you hack out some code to solve a problem with the goal being a better understanding of the problem's domain. This enables more accurate estimates and starts you off on a strong foot. Spikes are a very useful mechanism for getting started on a project or problem if you're completely clueless of how you want to approach. But spikes can also be dangerous, because once you've approached a problem with a spike it's hard to get the schema of the spike solution out of your head.

Recently, I was trying to implement some code that started up a server socket and continuously listened for new connections: dispatching new connection sockets to deal with the incoming connections. After the spike, I had something like this:

server = ServerSocket(5000)
server.initialize_and_listen_for_connections()
while True:
    conn_socket = server.accept_connection_and_return_socket()
    data = conn_socket.recv(4096)
    new_data = transform_data(data)
    conn_socket.send(new_data)


So with this simple code going, I had a pretty good feel for what needed to be done. Problems started to show up when I needed to write some production-quality code that achieved this functionality. I couldn't stop thinking about this solution and every test seemed difficult to write. Instead of test-driving my design, I was spike-driving my design, and this certainly isn't a good thing. Eventually, I paired up with my mentor and we were able to create some meaningful abstractions that were fairly simple to test and achieved the same level of functionality. In the future, I'll be wary of "spiking" something out, since it's so tempting to get stuck in a one-solution mindset.
