--- layout: article title: Naming Decisions date: '2013-11-02T22:01:00.001-07:00' author: aml tags: - apprenticeship modified_time: '2013-11-02T22:01:33.370-07:00' blogger_id: tag:blogger.com,1999:blog-1730093662423317857.post-6968817095679585217 blogger_orig_url: https://immutablearlandis.blogspot.com/2013/11/naming-decisions.html --- I wrapped up a new feature on my server today: you can now toggle the various features that you want enabled.
This is implemented via the command line where you set feature flags to true or false.
After finishing, I decided to refactor by taking a second look at the newly written code.
This code stood out to me:

public interface Toggler {
    Boolean isEnabled(String route);
}

You'll notice that this "Toggler" interface only has the "isEnabled" method defined on it.
To me, the name of this interface doesn't match up with what it's actually being used for. Were this my first time reading the code, I would think that something implementing the "Toggler" interface would give me the ability to, well, toggle! I didn't like the fact that the "Toggler" didn't actually let me toggle, so I decided to rename it.

The method "isEnabled" is interrogative, so I brainstormed names that reflected this querying ability. I thought "Directory" was a great name! In the same manner that you would look at a building directory to get a sense of what's in the building, the "Directory" lets you know which features are there.

Though I was pretty sold on this name, I remembered the context of the problem.
One of the features of the server is the ability to browse the local filesystem. Having the word "directory" in a name could make things fuzzy for the uninitiated. Reluctantly, I discarded this perfect name in search of another.

Finally, I settled on "Inventory". It's not as good as "Directory", but carries the same kind of meaning and accurately reflects its methods.
