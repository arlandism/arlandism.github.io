I knew absolutely nothing about git coming into my apprenticeship, but I've already learned a ton. I thought this would be a good opportunity to reflect and come up with a brief beginner-friendly tutorial.

Configuration

First, let's get into a directory that we want git to watch.


mkdir my_project_dir
cd my_project_dir
git init my_project_dir
Initialized empty Git repository in /Users/arlandislawrence/development/my_project_dir/.git/

We've initialized an empty git repository and we should get a status message confirming it. While we're configuring, we should probably create a .gitignore file. This is a settings file that tells git we don't want the stuff listed to show up.


touch .gitignore

We've created the file, but there's nothing in it yet. I'll leave that to you. Here's my .gitignore file if you're looking for examples. It's a good idea to ignore executables, binary files, text-editor generated stuff, etc.

Workflow

We've got git initialized. Let's do some stuff.

echo 'Gonna be a git pro!' > message.txt
git status
# On branch master
#
# Initial commit
#
# Untracked files:
#   (use "git add ..." to include in what will be committed)
#
# message.txt

That should create a message.txt file in your current working directory. You can run "git status" to see what's changed in your directory. Now run "git add" to add that changed file and run "git status" to see it staged.


git add message.txt
git status
# On branch master
#
# Initial commit
#
# Changes to be committed:
#   (use "git rm --cached ..." to unstage)
#
# new file:   message.txt
#

*Pro tip: Run "git add -u" to add every changed file that's already being tracked.

Now we have our changed file staged - it's ready to be committed. 
git commit -m "Add message.txt"
[master (root-commit) 700309b] Add message.txt
 1 file changed, 1 insertion(+)
 create mode 100644 message.txt

Awesome! Those are the very basics of using git. Those are pretty much the only commands I use on a daily basis, but there are quite a few others that one should be familiar with; I'm not familiar with them so I won't moonlight as a git pro.

I think I should make one important note before ending: Git and Github are not the same thing. When I started programming, I thought that git was just an API for Github. Git is a version control tool, while Github is a site for storing your git repositories. On the bright side, "pushing" your repositories to Github doesn't require much more effort.
